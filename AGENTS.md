# AGENTS.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update AGENTS.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in AGENTS.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No API backward compat**: Project has no external users yet — don't preserve old Python APIs, function signatures, or import paths. Rename, delete, and rewrite freely; no shims or re-export modules. DB schema changes still go through Django migrations as normal — existing installs must upgrade cleanly.

## Project Overview

OpenOutreach — self-hosted LinkedIn automation for B2B lead generation. Playwright + stealth for browser automation, LinkedIn Voyager API for profile data, Django + Django Admin for CRM (models owned by this project).

## Commands

```bash
# Docker
make build / make up / make stop / make logs / make up-view

# Local dev
make setup    # install deps + browsers + migrate + bootstrap CRM
make run      # run daemon
make admin    # Django Admin at localhost:8000/admin/

# Testing
make test / make docker-test
pytest tests/api/test_voyager.py   # single file
pytest -k test_name                # single test
```

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **`linkedin_cli`** (external package — [`eracle/linkedin-cli`](https://github.com/eracle/linkedin-cli), installed from PyPI as `linkedin-agent-cli` via `requirements/base.txt`, unpinned so builds get the latest auto-published release; import name `linkedin_cli`): Django-free library of LinkedIn platform mechanics (browser nav/login, Voyager API, profile/conversation scrape, connect/message/status/thread verbs), imports with no Django. Exposes a verb CLI — the installed `linkedin-cli` console command (or `python -m linkedin_cli.cli`): `session open/close`, `login`, `whoami`, `search`, `profile`, `status`, `connect`, `message`, `thread` — over a **bind+connect** transport — a session owner `browser.bind()`s the browser, clients `chromium.connect()`. **Output contract** (in the `cli.py` docstring so it travels with the package): each verb returns a result dict, rendered as a brief human summary by default or the full dict with `--json` (redirect with `>` to save; no `--out`); errors → stderr + non-zero exit. The daemon owns its browser in-process and consumes the library directly; launch/cookie-persistence glue stays in `openoutreach/linkedin/browser/launch.py`. **Auth is a page-state machine** (`page_state.py`: `classify_page` judges the page by URL *path* only, `@transition(when=,then=)` contracts enforce pre/post state raising `IllegalPageTransition`, `PageFlow` is the generic observe→act engine; `auth.py`: the login flow as declared `@auth_flow.transition` actions + `authenticate()`, shared by the CLI verb and the daemon — no duplicated login path). CRM/campaign/scheduling/ML stay in this project. (Extracted to its own repo + submodule; OpenOutreach consumes it as a dependency.)
- **Entry**: `manage.py` — stock Django management. `rundaemon` command (migrate → onboard → **email-setup nudge** → validate → task queue loop). `manage.py` with no args defaults to `rundaemon`. Onboarding logic in `onboarding.py`: `OnboardConfig` (pure dataclass), `missing_keys()`, `collect_from_wizard()`, single `apply()` write path. **Email-setup nudge** (`emails/nudge.py`, runs every launch after onboarding, **non-fatal/deferrable — never `sys.exit`**): until both a finder key and a working `Mailbox` exist it prompts on a TTY (or logs headless) the next step — `NO_FINDER` → capture the BetterContact key inline; `NO_MAILBOX` → paste the IceMail *Export Mailboxes* sheet (`emails/icemail.py` parses the `Email`/`Password` columns; each box is SMTP auth-checked via `emails/smtp.py` and stored `active=False` if auth fails). Copy is the GLF angle (`marketing/email-sequence.md`) filled with the user's own pipeline counts; no opt-out by design. The interactive wizard is self-contained — `onboarding_wizard.py` (questionary/prompt_toolkit step engine with back/skip/cancel) and `onboarding_prompts.py` (`SELF_HOSTED_QUESTIONS`), both vendored from the retired openoutreach-cli; no external `openoutreach` dependency. Docker `start` script handles Xvfb/VNC, then `exec python manage.py rundaemon`.
- **State machine**: `openoutreach/crm/models/deal.py:DealState` (OpenOutreach-owned `TextChoices`) — QUALIFIED → READY_TO_CONNECT → PENDING → CONNECTED → COMPLETED / FAILED, with an **email fork at QUALIFIED**: enrichment *routes* (it doesn't gate), and the route is an explicit FSM fork. A finder **hit** transitions QUALIFIED → **READY_TO_EMAIL** (a cheap, *ungated* FIFO send-queue, paced only by the per-box cap) → **EMAILED** on the single Layer-1 send (a quasi-terminal state that rests until a human sets an Outcome — no inbound reading at Layer 1). A **miss / finder-off / couldn't-run** leaves the deal QUALIFIED so the GP gate can promote it to READY_TO_CONNECT (its only door; the connection then harvests contact info on acceptance). The two fork states encode the one-shot guarantee in the state column — the email pool reads only READY_TO_EMAIL, so a deal sends exactly once and can never double-send. `Deal.state` is a CharField with `DealState` choices (no Stage model). The funnel is **not** in `linkedin_cli`: the library's connect/status verbs only *observe* three of these (QUALIFIED/PENDING/CONNECTED) off the LinkedIn UI and return them as plain strings over the CLI boundary, where the task handlers lift them via `DealState(value)`. The library keeps its own internal `linkedin_cli.enums.ProfileState` (those 3 UI states) — OpenOutreach no longer imports it. `Outcome` (converted/not_interested/wrong_fit/no_budget/has_solution/bad_timing/unresponsive/unknown) on Deal.outcome. `Lead.disqualified=True` = permanent exclusion. LLM rejections = FAILED Deals with wrong_fit outcome (campaign-scoped). **Contact capture on connection**: when a Deal first enters CONNECTED, `set_profile_state` fires a best-effort `Lead.capture_contact_info(session)` (the moment LinkedIn exposes a 1st-degree connection's email/phone) — failures are logged and never roll back the transition; `AuthenticationError` propagates. Lead email storage is one field per source: `Lead.contact_info` (raw LinkedIn overlay, written here) + `Lead.api_email` (enrichment-API result, writer lands with the finder).
- **Task queue**: `Task` model (persistent). Four types: `connect`, `check_pending`, `follow_up`, `email`. The first three are handled in `openoutreach/linkedin/tasks/`, `email` in `openoutreach/emails/tasks/send.py:handle_email`; shared signature `handle_*(task, session, qualifiers)`. Task rows are **lazy**: `payload = {"campaign_id": <id>}` only; the handler resolves the concrete target at execution time via a single eligibility query. Slot creation is centralized in `openoutreach/core/scheduler.py` — no other module inserts Task rows. The LinkedIn channels use **window planners** (`plan_connect_window`, `plan_follow_up_window`, `plan_check_pending_window`); each emits `1 immediate + (n-1) Poisson-spaced` slots over the next 24h working window. `working_seconds_in_window(start, end)` and `poisson_slot_times(now, n, horizon_hours)` are the spacing primitives (uniform order-statistic implementation, exactly `n` slots, mean spacing `T/(n+1)` in working time). Email has no anti-bot rhythm to fake, so it uses an **eager drain** instead: `flush_email_queue(session, campaign)` emits an *immediate* (un-spaced, un-ranked) slot for every READY_TO_EMAIL deal, capped by the pool-wide per-box daily headroom (`Mailbox.objects.remaining_today()`). State-transition hook `on_deal_state_entered(deal)` only stamps `Deal.next_check_pending_at` on PENDING (no Task row); all other transitions are no-ops. `reconcile(session)` runs on daemon idle: it recovers stale RUNNING tasks, then per campaign runs the window planners and the email flush, re-planning any type whose pending queue is empty. Active hours are a single contiguous daily window (`ENABLE_ACTIVE_HOURS`, `ACTIVE_START_HOUR`, `ACTIVE_END_HOUR`, `ACTIVE_TIMEZONE`) — no weekend skip. `CHECK_PENDING_DAILY_CAP` (default 100) bounds the per-cycle check_pending slot count. On 401 (`AuthenticationError`), the daemon calls `session.reauthenticate()` and marks the task FAILED; the planner picks the type up on the next idle cycle.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification. Per-campaign models stored in `Campaign.model_blob` (DB).
- **Config**: `SiteConfig` DB singleton (LLM_PROVIDER, LLM_API_KEY, AI_MODEL, LLM_API_BASE, FINDER_API_KEY — editable via Django Admin; `finder_api_key` is the BetterContact email-finder key, blank disables enrichment (see `emails/finder.py`); `llm_provider` chooses between OpenAI/Anthropic/Google/Groq/Mistral/Cohere/openai_compatible, `llm_api_base` only consulted when provider is `openai_compatible`), `conf.py:CAMPAIGN_CONFIG` (timing/ML defaults), `linkedin_cli/conf.py` browser constants (`BROWSER_*`, `HUMAN_TYPE_*`, `BROWSER_HEADLESS`), `conf.py` schedule constants (`ENABLE_ACTIVE_HOURS` flag, active hours/timezone/rest days), `conf.py` onboarding defaults (`DEFAULT_*_LIMIT`), `conf.py:FASTEMBED_CACHE_DIR` (persistent model cache, defaults to `<project>/.cache/fastembed/`), Campaign/LinkedInProfile models (Django Admin). `VOYAGER_REQUEST_TIMEOUT_MS` lives in `linkedin_cli/api/client.py` (constructor default on `PlaywrightLinkedinAPI`). `linkedin_cli/conf.py:DUMP_PAGES` (default `False`) — enable to save page HTML snapshots for fixture collection.
- **Lazy accessors**: `Lead.get_profile(session)` is a pure live Voyager scrape (no DB caching of the raw dict); `Lead.get_urn(session)` reads the `urn` column and falls back to a scrape; `Lead.get_embedding(session)` lazily scrapes + embeds on first access, then caches the 384-dim bytes on the row. `Lead.capture_contact_info(session)` scrapes the LinkedIn contact-info overlay (1st-degree only) and persists it raw to `Lead.contact_info` once — idempotent on non-null (even an empty result counts as "tried"). `Lead.resolve_api_email()` resolves a work email via the finder (`emails/finder.py`, URL-only, no session) and persists `Lead.api_email`. Tri-state: returns **True** on a hit (cached, never re-resolved — the qualify router transitions the Deal QUALIFIED → `DealState.READY_TO_EMAIL` via `set_profile_state`, routing it onto the email channel and out of the connect pool), **False** on a genuine miss (finder ran, found nothing — Deal stays QUALIFIED for the connect leg), **None** when the finder couldn't run (no key set, or service unreachable — `FinderUnavailable`, stays QUALIFIED to retry). A miss is free to retry — BetterContact bills only usable hits. Fired at qualification (`pipeline/qualify.py`, right after the QUALIFIED promotion) — enrichment only ever runs on qualified leads. `Lead.embed_from_profile(profile)` reuses an in-hand profile dict to skip the scrape (used by `create_enriched_lead`). `Lead.to_profile_dict()` returns a minimal `{lead_id, public_identifier, url, meta}` dict (no `profile` key). `AccountSession.campaigns` (cached_property, list). `AccountSession.self_profile` (cached_property, re-discovers via Voyager on first access per session — no DB cache).
- **Deal summaries**: `Deal.profile_summary` and `Deal.chat_summary` are lazy, mem0-style JSON fact lists built on demand and updated incrementally. `openoutreach/core/db/summaries.py` is the single boundary — `materialize_profile_summary_if_missing(deal, session)` fires on the first follow-up touch (one Voyager re-scrape per `(lead, campaign)` lifetime), `update_chat_summary(deal, new_messages, *, seller_name)` folds newly-synced ChatMessages into the summary via `reconcile_facts`, which routes new facts through mem0's UPDATE prompt to apply ADD/UPDATE/DELETE/NONE events (no naive append-and-dedup). **Only incoming (lead) messages reach fact extraction** — outgoing seller messages are filtered at the boundary so `chat_summary` stores facts about the lead, never the seller's pitch. A one-sided burst of outgoing messages short-circuits the LLM call entirely. **Identity binding (required):** `extract_facts`, `update_chat_summary`, and `reconcile_facts` all require a `seller_name` kwarg, sourced via `seller_name_from(session)` (first_name from `session.self_profile`, username fallback) and injected into the system prompt so the LLM stops inferring `the lead's name is Diego` from a lead reply like `"Hola Diego, gracias..."`. The mem0 reconcile prompt also receives the binding and is instructed to DELETE stored facts that describe the seller as if they were the lead — so previously-contaminated rows *should* be cleaned up on the next sync that produces a conflicting fact, though this is best-effort (mem0's vendored prompt is example-heavy and the cleanup hint is one prepended sentence). Dormant deals with no new messages stay contaminated until eager cleanup. The follow-up agent consumes `profile_summary + chat_summary + last 6 ChatMessage rows` instead of flat profile fields; its prompt (`follow_up_agent.j2`) carries the same `Me`→`self_name` binding under the verbatim-messages header. The fact-extraction prompt is vendored at `openoutreach/core/db/summaries.py:_FACT_EXTRACTION_PROMPT`; mem0's `DEFAULT_UPDATE_MEMORY_PROMPT` and `get_update_memory_messages` are vendored under `openoutreach/core/vendor/mem0/configs/prompts.py` (mirroring upstream paths so future syncs are a clean diff). No `mem0ai` runtime dependency — avoids qdrant/grpcio/sqlalchemy transitive bloat.
